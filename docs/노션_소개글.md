# 포트폴리오 RAG 에이전트 — 노션 소개글

---

## Preview

- **목적** : PDF/Word 포트폴리오를 업로드해 두면, 인사팀이 채팅으로 질문할 수 있는 RAG(검색 기반 생성) 에이전트 구축. Gradio + LangChain + FAISS + OpenAI 기반으로, Hugging Face Spaces 등에 배포 가능.
- **역할** : 3-way 라우터(RAG / NO_RAG / GENERAL) 설계, FAISS 벡터 검색·청킹·임베딩 파이프라인 구성, 스트리밍 답변·대화 요약 다운로드(MD/PDF)·키워드 통계 UI 구현, Faithfulness/Relevance 평가 후 재시도 로직 적용.
- **할 수 있는 것** : 포트폴리오 관련 질문은 문서 검색 후 답변, 인사·메타 질문은 검색 없이 응답, 무관 질문은 일반 LLM 응답. 참고 문단 접기 표시, 토큰 단위 스트리밍, 대화 요약·키워드 통계 제공.

---

## 1. 목적 및 설계 방식

- **목적**  
  지원자 포트폴리오(이력서·경력·프로젝트 등)를 문서로 넣어 두고, 인사 담당자가 자연어로 질문하면 **해당 문서만 참고해** 답하는 채팅형 에이전트를 만드는 것.
- **설계 방식**  
  - **질문 분류(3-way 라우터)**  
    매 질문마다 LLM으로 한 번에 세 가지 중 하나로 분류한다.  
    - **RAG** : 포트폴리오 문서를 검색해서 구체적으로 답해야 하는 질문(경력, 프로젝트, 강점 등).  
    - **NO_RAG** : 포트폴 맥락이지만 문서 검색 없이 답할 수 있는 질문(안녕, 너 누구야?, 뭐 할 수 있어? 등).  
    - **GENERAL** : 포트폴리오와 무관한 질문(날씨, 일반 상식 등).  
    대화 히스토리(최근 N턴)를 라우터에 넘겨 맥락 있는 분류를 하도록 설계했다.  
  - **RAG 경로**  
    질문 문장을 임베딩해 FAISS에서 유사 문단을 k개 검색 → 이어 붙여 context로 만든 뒤, “아래 참고 자료만 사용해 답하라”는 프롬프트와 함께 OpenAI에 넘겨 답변 생성.  
  - **품질 보정**  
    설정으로 켜 두면, 생성된 답변을 LLM으로 Faithfulness(참고 문단 충실도)·Relevance(질문 관련성) 평가하고, 점수가 기준 미만일 때만 **k를 늘려 1회 재검색·재생성**하는 “평가 후 재시도”를 적용한다.

## 2. 업무 과정 (활용 도구·데이터 흐름)

- **활용 도구**  
  - **Gradio** : 채팅 UI, 예시 질문, 대화 초기화, 요약 다운로드 버튼, 통계·키워드 표시.  
  - **LangChain** : Retriever, Prompt, LLM 체인 구성 및 스트리밍.  
  - **FAISS** : 포트폴리오 청크의 벡터 인덱스 저장·검색.  
  - **HuggingFace Embeddings** : 한국어 임베딩(`jhgan/ko-sroberta-multitask`)으로 청크·질문 벡터화.  
  - **OpenAI** : 라우터 분류용·답변 생성용 LLM(gpt-4o-mini 등).  
  - **PyPDFLoader / Docx2txtLoader** : PDF·Word 로드.  
  - **reportlab** : 요약 다운로드 시 PDF 생성 시도(한글 실패 시 TXT로 대체).

- **데이터 흐름 구조**  
  1. **인덱스 빌드(사전 1회)**  
     `data/portfolio/`의 PDF·DOCX → 로드 → RecursiveCharacterTextSplitter(600자/100자 겹침) 청킹 → 임베딩 → FAISS로 `index/`에 저장.  
  2. **앱 기동**  
     Gradio UI 기동 시 `index/`의 FAISS 인덱스는 첫 질문 시점에 로드해 Retriever·Chain을 만들고 이후 재사용.  
  3. **질문 입력 시**  
     사용자 입력 + 현재 채팅(messages) → 내부적으로 `pairs`로 변환 → `get_answer_stream(question, pairs)` 호출.  
  4. **라우팅**  
     `_route_question(question, history_pairs)`로 RAG / NO_RAG / GENERAL 결정.  
  5. **RAG 분기**  
     Retriever로 질문과 유사한 문단 k개 검색 → context 문자열 생성 → 스트리밍 체인으로 OpenAI에 전달 → 토큰 단위로 yield.  
  6. **NO_RAG / GENERAL 분기**  
     검색 없이 시스템 프롬프트만 구분해 LLM 호출 후 스트리밍으로 yield.  
  7. **UI 갱신**  
     매 chunk마다 (new_pairs, 통계, 키워드)를 yield하고, Gradio Chatbot을 messages로 변환해 갱신. 마지막에 참고 문단을 `<details>` 접기로 붙여 표시.

## 3. 구조 (주요 컴포넌트·액션)

| 구분 | 내용 |
|------|------|
| **config.py** | 경로(`data/portfolio`, `index/`), 청킹(600/100), 검색 k·재시도 k, 임베딩/LLM 모델, 평가 재시도 on/off·최소 점수 등 한곳에서 관리. |
| **scripts/build_index.py** | 포트폴리오 로드 → 청킹 → 임베딩 → FAISS 저장. 앱은 이 결과물(`index/`)만 읽음. |
| **app/rag.py** | 3-way 라우터(`_route_question`), Retriever 로드(`_load_retriever`), RAG/NO_RAG/GENERAL 분기, `get_answer`(비스트리밍·평가 후 재시도 포함), `get_answer_stream`(스트리밍). |
| **app/rag_eval.py** | (질문, 참고 문단, 답변)을 LLM으로 평가해 Faithfulness/Relevance 점수·이유 반환. rag.py에서 재시도 조건 판단 시 사용. |
| **app/app.py** | Gradio UI 구성(build_ui), 전송/초기화/요약 다운로드 이벤트, messages↔pairs 변환, 통계·키워드·예시 질문, 스트리밍 응답 처리. |

- **액션 요약**  
  - 라우터 프롬프트로 RAG/NO_RAG/GENERAL 한 번에 구분, 대화 히스토리 반영.  
  - RAG 시 FAISS에서 k개 문단 검색 → context → 프롬프트 → OpenAI 스트리밍.  
  - (옵션) Faithfulness/Relevance 기준 미달 시 k 늘려 1회 재검색·재생성.  
  - 답변 + 참고 문단 접기, 대화 요약(MD/PDF 또는 TXT), 질문 키워드(경력·강점·프로젝트 등) 통계 제공.

---

## 4. 한계·참고 사항

- 인덱스는 **문서를 바꾼 뒤** `build_index.py`를 다시 실행해야 반영된다.  
- 라우터·답변 생성 모두 OpenAI를 사용하므로, 라우터를 더 가벼운 모델로 두면 비용·지연을 줄일 수 있다.  
- 배포 시 Hugging Face Spaces에서는 **Private** 권장(개인정보 고려).  
- 평가 후 재시도는 스트리밍 경로에는 적용하지 않고, 비스트리밍 `get_answer`에만 적용된 구조다.
