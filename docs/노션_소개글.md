# 1. 목적 및 설계 전략

## 🎯 목적

PDF/Word 또는 **Google Drive (Docs·Sheets·PDF)** 포트폴리오를 인덱싱하여  
인사 담당자가 **채팅으로 질문하면 답하는 RAG(검색 기반 생성) 에이전트** 구축

---

## 🧠 핵심 설계: 3-Way 라우터 구조

모든 질문은 LLM을 통해 아래 3가지 중 하나로 분류됩니다.

### 1️⃣ RAG

- 포트폴리오 문서 검색이 필요한 질문
- 예: 경력, 프로젝트 상세, 사용 기술, 강점 등

### 2️⃣ NO_RAG

- 포트폴 맥락이지만 문서 검색 없이 답 가능한 질문
- 예: "안녕하세요", "너 뭐 할 수 있어?"

### 3️⃣ GENERAL

- 포트폴리오와 무관한 일반 질문
- 예: 날씨, 상식 등

> 최근 N턴 대화 히스토리를 함께 전달하여 문맥 기반 분류 수행

---

## 💬 대화 맥락(히스토리) 관리 방식

- **전체 대화**는 Gradio Chatbot(UI)에만 보관하고, 사용자가 질문할 때마다 그 히스토리를 백엔드로 넘깁니다. 별도 메모리 DB나 세션 저장은 쓰지 않습니다.
- **라우터(분류)**
    - 최근 5턴을 터널당 최대 200자로 잘라 문자열로 만든 뒤, 라우터 프롬프트의 `<recent_conversation>`에 넣음  
    → "아까 경력 말했는데, 그럼 프로젝트는?"처럼 짧은 후속 질문도 맥락을 보고 RAG로 분류 가능

---

## 🔁 RAG 동작 흐름

1. **Query Expansion**  
   질문만 LLM에 넣어 재작성(Few-shot). 단일 질문 → 1줄, 포괄적 질문 → 2~4개 하위 질문. [원본, 재작성1, 재작성2, ...] 리스트 생성.
2. **쿼리별 Hybrid Search**  
   각 쿼리마다 Dense(FAISS) top-k + Sparse(BM25/Kiwi) top-k 검색 → RRF로 병합해 쿼리당 N개 문서 랭킹 생성.
3. **다중 랭킹 RRF 병합**  
   여러 쿼리의 랭킹을 다시 RRF로 합쳐 상위 N개 선정.
4. **Reranker**  
   Cross-Encoder로 재순위화하여 상위 4개만 선별(precision 확보).
5. **Context 구성**  
   선별된 문단을 context로 붙임.
6. **LLM 호출**  
   "아래 참고 자료만 사용해 답하라" 프롬프트와 함께 LLM 호출, 토큰 단위 스트리밍 응답 생성.

# 2. 데이터 흐름 및 아키텍처

## 🛠 사용 기술

- **Gradio**  
  채팅 UI, 예시 질문, 대화 초기화, 요약 다운로드 버튼, 통계 표시

- **LangChain**  
  Retriever·Prompt·LLM 체인, 스트리밍 연결

- **FAISS**  
  벡터 인덱스 저장 및 유사도 검색 (Dense)

- **BM25 + Kiwi**  
  Sparse 검색용. Kiwi 형태소로 토큰화 후 BM25 인덱스, 쿼리당 top-k 검색

- **RRF (Reciprocal Rank Fusion)**  
  Dense·Sparse 결과 및 다중 쿼리 랭킹 병합

- **HuggingFace Embeddings**  
  `jhgan/ko-sroberta-multitask` 한국어 임베딩

- **OpenAI**  
  라우터·Query Expansion·답변 생성용 LLM (예: gpt-5, gpt-5-mini)

- **문서 로더**  
  로컬: PyPDFLoader, Docx2txtLoader  
  구글: GoogleDriveLoader (Docs·Sheets·PDF)

- **reportlab**  
  PDF 요약 다운로드 시 사용 (한글 실패 시 TXT 대체)

- **sentence-transformers CrossEncoder**  
  BAAI/bge-reranker-v2-m3 — 검색 결과 정밀 재순위화용 다국어 Cross-Encoder

# 4. 주요 기능 정리

- **3-way 질문 라우팅**: 문맥 기반 RAG 여부 자동 판단
- **Query Expansion**: 질문만 LLM 재작성, 포괄적 질문은 2~4개 하위 질문으로 분할 후 각각 검색·RRF 병합
- **Hybrid Search**: FAISS(Dense) + BM25/Kiwi(Sparse) → 쿼리당 RRF → 다중 쿼리 RRF 병합
- **Reranker 재순위화**: Cross-Encoder로 최종 상위 4개만 LLM에 전달
- **품질 평가 후 재시도**: Faithfulness/Relevance 기준 미달 시 k 늘려 1회 재검색·재생성
- **참고 문단 접기 표시**: 답변 하단 `<details>` 형태로 제공
- **대화 요약 다운로드**: Markdown / PDF (한글 실패 시 TXT)
- **질문 키워드 통계**: 경력 / 강점 / 프로젝트 등 자동 집계

# 5. 한계 및 개선 포인트

- 문서 변경 시 `build_index.py` 재실행 필요
- 라우터·Query Expansion·답변 생성 모두 OpenAI 사용 → 비용 증가 가능성  
  - 개선안: 라우터/확장은 경량 모델로 고정
- Reranker·Kiwi 최초 로드 시 수 초 소요 (이후 싱글톤 재사용으로 영향 완화)
- BM25 인덱스 없으면 Dense만 사용(자동 폴백)
