### app/rag.py
| 함수 | 기능 |
|------|------|
| **_format_docs(docs)** | Document 리스트를 `"\n\n---\n\n"`로 이어 붙인 하나의 context 문자열로 만듦. |
| **_strip_html(text)** | HTML 태그 제거 후 앞뒤 공백 정리. |
| **_pairs_to_messages(pairs, max_turns)** | `[[user,bot],...]` → LangChain HumanMessage/AIMessage 리스트. 최근 `max_turns`턴만 사용. |
| **_load_retriever(k)** | FAISS 인덱스 로드 후 Retriever 반환. `k` 미지정 시 config.RETRIEVE_K, 재시도 시 RETRIEVE_K_RETRY 사용. |
| **_is_portfolio_related(question)** | LLM으로 "질문이 포트폴리오 관련인가?" 판별. YES면 True, NO면 False. API 없으면 True. |
| **_generate_with_context(question, context, chat_history)** | 이미 만든 context로 RAG 스타일 답변 생성. **평가 후 재시도** 시 사용. |
| **get_chain()** | Retriever + (question→context→prompt→LLM→문자열) 체인 생성. 체인과 retriever 둘 다 반환. |
| **get_answer(question, history_pairs)** | 포트폴 관련이면 RAG(검색→답변), 무관이면 LLM만. 필요 시 평가 후 재시도( k 늘려 1회 재검색·재생성). `(answer, source_docs)` 반환. |
| **get_answer_stream(question, history_pairs)** | 위와 같은 라우팅이지만 답변을 **스트리밍**으로 yield. 재시도 없음. |

### app/rag_eval.py
| 함수 | 기능 |
|------|------|
| **_parse_eval_output(text)** | LLM 평가 출력에서 FAITHFULNESS/RELEVANCE 점수(1~5)와 REASON_F/REASON_R 한 줄 이유를 파싱해 dict 반환. |
| **evaluate_response(query, context, answer, model)** | (질문, 참고 문단 문자열, 답변)을 LLM으로 평가해 Faithfulness·Relevance 점수와 이유 dict 반환. |
| **evaluate_response_from_docs(query, source_docs, answer, model)** | source_docs를 context 문자열로 합친 뒤 `evaluate_response` 호출. |

### app/app.py

실행 시 `build_ui()`로 만든 demo에 `launch()` 호출.

## 1. 전체 그림 (한 줄 요약)

**사용자가 질문** → **Gradio 앱**이 입력을 받아 → **RAG(rag.py)**가 포트폴리오에서 관련 문단을 **검색**하고 → **OpenAI**가 그 문단을 참고해 **답변 생성** → **채팅창**에 표시.  
인덱스는 **미리** `scripts/build_index.py`로 만들어 두고, 앱은 그 **인덱스 파일만 읽어서** 검색한다.

---

## 2. 실행 순서 (두 단계)

| 단계 | 언제 | 뭘 하는지 |
|------|------|-----------|
| **1) 인덱스 빌드** | 포트폴리오 PDF/Word를 넣거나 바꾼 뒤 **한 번** | `uv run python scripts/build_index.py` → `data/portfolio/` 문서를 읽어서 잘게 나누고, 벡터로 만들어 `index/` 에 저장 |
| **2) 앱 실행** | 대화하고 싶을 때 | `uv run python app.py` → Gradio 웹 UI가 뜨고, 여기서 질문하면 RAG가 `index/` 를 읽어서 답변 |

즉, **문서가 바뀌면 1번 다시 실행**, **대화만 할 때는 2번만** 실행하면 된다.

---

## 3. config.py — 설정 한곳에

- **경로**: `data/portfolio/`(원본 문서), `index/`(FAISS 인덱스)
- **청킹**: 한 조각 600자, 100자 겹침
- **검색**: 질문당 가져올 문단 개수 4개(`RETRIEVE_K`), 평가 재시도 시 8개(`RETRIEVE_K_RETRY`)
- **평가 후 재시도**: `EVAL_RETRY_ENABLED`, `EVAL_MIN_FAITHFULNESS`, `EVAL_MIN_RELEVANCE` (낮으면 k 늘려 1회 재검색·재생성)
- **모델**: 임베딩 `jhgan/ko-sroberta-multitask`, 답변 생성 `gpt-4o-mini`

나중에 숫자나 모델만 바꾸고 싶으면 여기만 수정하면 된다.

---

## 4. scripts/build_index.py — 인덱스 만드는 단계

```
data/portfolio/ (PDF, Word)
    ↓
1) 로드 (PyPDFLoader, Docx2txtLoader)
    ↓
2) 청킹 (RecursiveCharacterTextSplitter, 600자/100자 겹침)
    ↓
3) 임베딩 (jhgan/ko-sroberta-multitask → 문장을 숫자 벡터로)
    ↓
4) FAISS에 저장 → index/ (index.faiss, index.pkl)
```

- **로드**: 폴더 안의 PDF·DOCX를 LangChain 로더로 열어서 텍스트로 만든다.
- **청킹**: 긴 문서를 600자 단위로 자르고, 100자 겹치게 해서 “문단” 단위로 쓴다.
- **임베딩**: 한국어 문장을 벡터로 바꿔서 “의미가 비슷한 문장 = 벡터가 가깝다”가 되게 한다.
- **FAISS**: 그 벡터들을 `index/` 에 파일로 저장해 두고, 나중에 “질문 벡터와 가장 가까운 문단”을 빠르게 찾을 수 있게 한다.

**한 번 돌려두면** 앱은 이 `index/` 만 읽고, 매번 PDF/Word를 다시 읽지 않는다.

## 5. app/rag.py — RAG가 어떻게 돌아가는지

### 5.1 한 번만 로드하는 것

- **FAISS 인덱스**: `index/` 에서 `index.faiss` + `index.pkl` 읽기 (같은 임베딩 모델 사용).
- **Retriever**: “이 질문과 비슷한 문단 **4개**를 찾아라”라고 FAISS에 물어보는 객체.
- **Chain**: “찾은 문단(context) + 사용자 질문(question)”을 **프롬프트**에 넣고 → **OpenAI**에 보내서 → **한국어 답변** 한 줄 받는 파이프라인.

`_chain`, `_retriever` 는 앱이 처음 질문을 받을 때 한 번만 만들고, 그 다음부터는 재사용한다.

### 5.2 질문 들어왔을 때 흐름 (get_answer)

1. **_is_portfolio_related(question)**  
   - LLM으로 포트폴리오 관련 여부 판별. 무관하면 NO_RAG로 LLM만 호출 후 (answer, []) 반환.
2. **포폴 관련이면 retriever.invoke(question)**  
   - 질문 문장을 임베딩해서, FAISS에서 가장 가까운 문단 **4개**를 가져온다.
3. **context**  
   - 그 4개 문단을 `"\n\n---\n\n"` 로 이어 붙여서 하나의 “참고 자료” 문자열로 만든다.
4. **프롬프트 + LLM**  
   - 시스템 메시지: “아래 [참고 자료]만 사용해서 답하라, 없으면 없다고 하라, 한국어로 간결하게.”  
   - 사용자 메시지: `question` (사용자가 쓴 질문)
5. **평가 후 재시도 (옵션)**  
   - EVAL_RETRY_ENABLED 이고 source_docs 가 있으면 Faithfulness/Relevance 평가. 최소 점수 미만이면 1회만 k 를 RETRIEVE_K_RETRY 로 늘려 재검색 후 _generate_with_context 로 재생성.
6. **반환**  
   - `(답변 문자열, 출처 문단 4개)` 를 앱에 넘긴다.

### 5.3 스트리밍 (get_answer_stream)

- **같은 검색**: 위와 같이 먼저 4개 문단을 찾고, `context` 를 만든다.
- **다른 점**: 답변을 한 번에 받지 않고, `stream_chain.stream(...)` 으로 **토큰 단위로** 받아서, 누적 문자열을 계속 `yield (full, source_docs)` 로 넘긴다.
- **앱 쪽**: 이걸 받아서 채팅창에 “글자가 조금씩 늘어나는” 식으로 보여준다.

## 6. app/app.py — Gradio UI와 이벤트

### 6.1 화면 구성

- **채팅창 (Chatbot)**  
  - Gradio 6은 메시지를 `[{role: "user"|"assistant", content: "..."}]` 형식으로만 받는다.  
  - 우리는 내부적으로 `[[user, bot], [user, bot], ...]` (pairs) 로 다루고, **경계에서만** `_to_messages` / `_from_messages` 로 변환한다.
- **통계**  
  - 대화 턴 수, 총 글자 수를 히스토리에서 계산해서 `_format_stats` 로 표시.
- **키워드 통계**  
  - 사용자 **질문** 텍스트에서 “경력, 강점, 프로젝트, 역할…” 등이 몇 번 나오는지 세서 막대처럼 보여준다.
- **예시 질문 (gr.Examples)**  
  - 클릭하면 해당 문장이 입력창에 들어가고, 전송하면 위 RAG 흐름이 그대로 돌아간다.

### 6.2 질문 전송 시 (전송 버튼 / Enter)

1. 사용자 입력 `message` + 현재 채팅 `chat_value`(messages 형식) 를 받는다.
2. **`_from_messages(chat_value)`** → 내부용 **pairs** 로 바꾼다.
3. **`_submit_stream(message, pairs)`** 호출:
   - API 키·인덱스 존재 여부 검사.
   - **`get_answer_stream(message)`** 로 스트리밍 답변을 받으면서,  
     `(new_pairs, clear_txt, stats, keywords)` 를 여러 번 **yield**.
4. **`_to_messages(new_pairs)`** 로 다시 messages 로 바꿔서 **Chatbot** 에 넣어 준다.
5. 그때마다 **통계**, **키워드**도 같이 갱신한다.
6. 마지막 yield 에서는 “참고 문단” 블록까지 붙은 최종 답변을 넣어 준다.

즉, **입력 → pairs 변환 → RAG(스트리밍) → 매번 pairs 갱신 → messages로 변환 → 화면 갱신**.

### 6.3 대화 초기화

- 채팅을 **첫 인사 한 번만 남긴 상태**로 되돌린다.  
- `_to_messages([[None, FIRST_MESSAGE]])` 로 Chatbot 값을 바꾸고, 통계·키워드를 0/비움으로 갱신한다.

### 6.4 요약 다운로드 (MD / PDF)

- Chatbot 의 **messages** 를 **`_from_messages`** 로 **pairs** 로 바꾼다.
- **전체 대화 텍스트**를 만들고, 필요하면 **OpenAI**로 “3~5문장 요약”을 만들어서:
  - **MD**: 요약 + 전체 대화를 마크다운 파일로 저장 → 그 경로를 `gr.File` 에 넘겨서 다운로드 링크가 나오게 한다.
  - **PDF**: 같은 내용을 reportlab으로 PDF로 만들려고 시도하고, 실패하면(한글 폰트 등) 같은 내용을 .txt 로 저장해서 그 경로를 넘긴다.

## 7. 데이터 흐름 요약

```
[사용자] 질문 입력
    ↓
[app.py] message + chat_value(messages)
    → _from_messages → pairs
    → _submit_stream(message, pairs)
         ↓
[rag.py] get_answer_stream(question)
    → retriever.invoke(question)  →  FAISS에서 유사 문단 4개
    → context 문자열
    → stream_chain.stream({question, context})  →  OpenAI가 토큰 단위 답변
    → yield (full, source_docs)
         ↓
[app.py] (new_pairs, stats, keywords) 여러 번 yield
    → _to_messages(new_pairs) → Chatbot 갱신
    → 통계·키워드 영역 갱신
         ↓
[사용자] 채팅창에 답변이 차례로 보임 + 참고 문단(접기)
```

**인덱스 빌드**는 이 흐름과 분리되어, `build_index.py` 가 **문서 → 청킹 → 임베딩 → FAISS 저장** 한 번 해 두면, 앱은 **저장된 index/ 만 읽어서** 위 RAG가 동작한다.

이렇게 돌아가도록 코드가 짜여 있다.

## 개선 검토 및 반영 사항

### 반영한 개선 (rag.py)
- **RAG 경로 retriever 이중 호출 제거**: `get_answer`에서 검색 1회만 수행하고 `_generate_with_context`로 답변 생성. 이전에는 체인 내부 assign과 별도로 `_retriever.invoke`를 한 번 더 호출하고 있었음.
- **NO_RAG/GENERAL 중복 제거**: `_invoke_no_rag`, `_stream_no_rag` 헬퍼로 시스템 프롬프트만 바꿔서 호출하도록 통합.
- **라우터 프롬프트 안전 치환**: 사용자 입력(질문·히스토리)에 `{`, `}`가 있으면 `format()` 오류/주입 가능성이 있어, `_safe_format()`으로 치환 전 이스케이프 처리.

### 추가로 고려할 수 있는 개선
- **app.py와 rag.py의 `_strip_html` 중복**: 공통 유틸 모듈로 분리하면 유지보수에 유리.
- **예외 로깅**: `_route_question` 등에서 `except` 시 `return "RAG"`만 하고 로그를 남기지 않음. 디버깅 시 `logging.exception` 등 추가 권장.
- **config**: `INDEX_DIR` 존재 여부를 앱 기동 시 한 번 검사해 안내 메시지를 미리 보여주는 선택지.
- **문서**: 라우터가 3-way(`_route_question`)로 변경된 점을 위 표(rag.py)에 반영하면 좋음.
